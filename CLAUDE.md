# CLAUDE.md — Книга памяти (seed)

Это **seed-проект**: форкнул → запустил `claude` → получил готовую книгу памяти о
своих предках-фронтовиках (обложка, карты, иллюстрации, главы, PDF). Каждый
параметр — фамилия, село, автор, регион — берётся из **`book.config.yml`**.
Хардкода нет нигде. Референс-реализация (собранная книга Кагановых) — по ссылке в
`book.config.yml → reference_book_url`.

> **Первым делом в новой сессии:** прочитай `book.config.yml`, затем
> `wiki/index.md`, затем `MEMORY.md` (если есть), затем этот файл целиком.
> Если `book.config.yml` не заполнен (пустой `subject.surname`) — предложи
> пользователю команду `/start`.

Паттерн знаний — **LLM-wiki** Андрея Карпаты
([gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)): не
пересобирать знание через RAG каждый раз, а постепенно строить структурированный
перекрёстно-связанный набор markdown-страниц между сырыми источниками и книгой.
Знание **накапливается**, не теряется в истории чатов.

## 1. Файловая структура (пять чистых сущностей)

```
book.config.yml      ЕДИНЫЙ конфиг: фамилия, село, автор, регион, режим
CLAUDE.md            этот файл — мозг проекта (грузится автоматически)
MEMORY.md            индекс заметок об этом проекте/пользователе (необязательно)

.claude/agents/      10 субагентов (параметризованы из book.config.yml)
.claude/commands/    слэш-команды = стадии конвейера

sources/             RAW, НЕИЗМЕНЯЕМОЕ
  <pdf>              скан родословной (если есть)
  document/page_NN.md  постраничный OCR этого PDF
  scans/             доп. сканы от пользователя

wiki/                ЗНАНИЕ (llm-wiki)
  index.md  log.md
  people/ places/ units/ events/ families/

book/                РУКОПИСЬ
  _master/           outline.md introduction.md conclusion.md
                     interlude-<topic>.md sources.md STYLE-RULES.md dvory.yml
  chapters/<slug>/   facts.md  context.md  draft.md  review.md  maps.json
  illustration/      candidates.md  <slug>.md  _contact/

assets/              ВСЕ бинарники, по типам (HTML и картинки НЕ вперемешку)
  documents/<slug>/  сканы архивных документов ЦАМО + MANIFEST.md (researcher;
                     первоисточник фактов/портретов; в книгу НЕ вставляются)
  portraits/         лица людей из исходного PDF
  photos/            одобренные документальные фото мест/операций
  photos-candidates/ предложенные, ждут гейта автора
  cover/  fonts/  geo/

build/               КОД + транзит
  build_maps.py  build_trees.py  build_book.py  clip_geo.py
  extract_annots.py  ← вычитка: аннотации PDF → PROOF-файл (стадия /proof)
  theme.css  RULES.md
  .out/              ← весь генерат карт/деревьев/manuscript (в .gitignore)

dist/book.pdf        ИТОГ
```

**Дисциплина чистоты (это была главная боль исходного проекта):**

- Бинарники — только в `assets/`, разложены по типам. Никогда не сваливать
  `*.html` и `*.png` в корень или вперемешку.
- Весь генерат карт/деревьев/`manuscript.html` идёт в `build/.out/`
  (gitignored). В корне `build/` — только код.
- **Одна `draft.md` на главу.** Никаких `draft-v1..v9`. Версии = git-история.
  Ревью — `review.md` (перезаписывается). Это git-нативно и убирает россыпь.
- Карт-конфиг — `book/chapters/<slug>/maps.json` (по файлу на главу, без
  merge-конфликтов), а не один общий питоновский список.
- `sources/` = сырьё (immutable), `wiki/` = знание, `book/` = рукопись,
  `assets/` = медиа, `build/` = код, `dist/` = выход. Шесть слов — вся карта.

## 2. ЖЕЛЕЗНЫЕ ПРАВИЛА (нарушать нельзя — выстраданы правками)

Это ядро продукта. Действуют для **всех** агентов и для оркестратора.

1. **НОЛЬ ВЫМЫСЛА.** Никакого внутреннего монолога («он подумал», «ему
   казалось», «он представил», «вдруг», «вдалеке грохотало»), эмоций без
   источника, реконструкций «как было» («утром, когда туман…»), выдуманных
   диалогов. Лучше сухо и коротко, чем красиво и придумано.
2. **«СЕМЕЙНОЙ ПАМЯТИ» НЕТ.** Источники только: родословная/исходный документ
   (часто ~1 строка на человека) + архивы (ЦАМО/pamyat-naroda) + то, что
   пользователь **явно** сообщил (зафиксировано в `facts.md` как «уточнение
   родственника, дата»). Запрещено «по семейной памяти…», «семья десятилетиями
   считала…», «возможно, между 1943 и 1945…» без конкретной строки-источника.
3. **БЕЗ ДРАМЫ.** Расхождение источников — сухой факт, не «главный конфликт»,
   не «опровергает», не «закрывает вопрос». Просто записать обе версии.
4. **БЕЗ ПОВТОРОВ.** Перед добавлением — перечитать всю главу. Каждый факт —
   один раз. Сильный кадр («отец не узнал о смерти сына») — один раз. Цитату
   можно **развить**, нельзя пересказать своими словами рядом. Мета-контекст
   (две ветки, родство) — во введении, в главах не повторять.
5. **БЕЗ МЕТА-КОММЕНТАРИЕВ ПРАВОК В ТЕКСТЕ КНИГИ.** «В прежних версиях…»,
   «как мы выяснили выше», «исследование показало» — вон.
6. **БЕЗ АНГЛИЦИЗМОВ, НЕРАСШИФРОВАННЫХ СОКРАЩЕНИЙ, МЕТОДОЛОГИИ И URL В
   ТЕКСТЕ.** `дислокация`→расположение, `коммуникации`→связь, `эпизод`→случай,
   `локация`→место, `OCR/URL/http/домены`→вон. `БВ`→«пропал без вести», `СД`→
   «стрелковая дивизия» (расшифровать при первом упоминании). Про поиск по
   опечаткам фамилии — один раз в предисловии, в главах ни слова. «20 тысяч
   футов» — мерять в метрах.
7. **СЛОВО «КЛАН» ЗАПРЕЩЕНО.** Только «семья», «ветка», «фамилия». «Узел»
   (о семьях) — тоже нельзя.
8. **КАРТЫ:** никогда не рисовать линию фронта (нет надёжных данных — «лучше
   нет карты, чем неправдивая»); никогда не выносить дом/призыв/родное село на
   боевую карту (ломает масштаб); показывать те города, что упомянуты в тексте;
   старое→новое имя города в скобках; подписи перекрывают фон полностью (halo).
9. **ИСТОЧНИКИ — ЕДИНЫМ СПИСКОМ В КОНЦЕ КНИГИ**, нумерованные сноски `[^N]` по
   главам, без URL и доменов в тексте сносок печатной книги.
10. **НЕ ВЫРЕЗАТЬ ИНФОРМАЦИЮ ПРИ ПРАВКЕ.** Правка = убрать повтор, сохранить
    всю фактуру. Потерял источник, на котором держался кусок, — найти заново,
    а не удалять кусок.
11. **АРХИВНАЯ ГЛУБИНА** (depth=standard/exhaustive): открыть каждый документ
    карточки, каждую страницу многостраничного документа, подвигать
    переключатель страниц. >2000 страниц = это уже чужие люди, не подшивать.
    Всегда сметка опечаток фамилии. Сначала зафиксировать находки, потом новый
    запрос. Проверять вывод субагента, прежде чем доверять (класс ошибок:
    «справка лежала рядом с приказом → приписал чужую часть»).
12. **chrome-devtools MCP** для pamyat-naroda (WebFetch там 403/CSRF; Stagehand
    не использовать). Гео-фенс (goskatalog/rgakfd/архивы РФ недоступны
    серверно) → попросить пользователя открыть ссылку у себя и вставить.
    Не закрывать браузер посреди задачи, если пользователь читает.
13. **ГЕЙТЫ ЧЕЛОВЕКА.** Иллюстрации: PROPOSE → автор отметил → APPLY (никогда
    не вставлять молча). PDF пересобирается **только** по явной команде
    «пересобери». Капча — остановиться и попросить пользователя.
14. **АВТОР-ВСТУПЛЕНИЕ — ПО ИНТЕРВЬЮ, НЕ ВЫДУМАНО.** Биографию, мотивы,
    поездки, чувства автора брать только из `book.config.yml → author`. Честная
    AI-оговорка + контакт для уточнений. Книга — «снимок памяти на <дата>, не
    итог», открытый и пополняемый документ. Если `author.is_writer:false` —
    добавить «иначе бы я не написал — я инженер, а не писатель».
15. **БЕЗ AI-ОБРАБОТКИ ФОТО.** Никакой генерации, «реставрации»-дорисовки,
    апскейла, колоризации-как-оригинала. Документальная книга = ноль
    галлюцинаций в изображениях тоже.
16. **«ПСЕВДОНИМЫ» (эпитеты).** У многих героев похожие имена. Каждому — стойкий
    эпитет (Учитель, Авиатехник, Снайпер). В чужих главах ссылаться через
    эпитет+имя, не «брат такого-то с соседнего участка» (зависит от другого
    человека и непонятно). Реестр — `book/_master/dvory.yml → epithets`.
17. **ЗАЧИН ГЛАВЫ — КАНОН:** `# Имя Отчество. Подзаголовок` → инлайн
    мини-дерево двора → лёгкая проза «точка отсчёта». Без курсивного эпиграфа,
    без пересказа родословной, без «Пролог/Глава/Часть/Эпилог» — у разделов
    содержательные имена («Призыв», «Извещение», «Что было после»).

## 3. Конвейер (карта стадий = слэш-команды)

```
/start       онбординг (вопросы) → заполняет book.config.yml → дальше по mode
/ingest      sources/<pdf> → sources/document/page_NN.md → список людей → wiki-стабы
/genealogy   genealogist: wiki/families/*, book/_master/dvory.yml (ветки/дворы),
             реестр эпитетов, логические конфликты дерева
/research    на каждого: pamyat-naroda (chrome-devtools, сметка опечаток,
             «прокликать каждую страницу», скачать сканы) → facts.md + context.md
/conceive    book-director: outline.md, introduction, conclusion, врезки,
             STYLE-RULES.md, sources.md (берёт дерево+эпитеты от genealogist)
/write       chronicler: chapters/<slug>/draft.md (тон Алексиевич/Гранин, ноль вымысла)
/review      reviewer (факты+стиль+повторы внутриглавно) → book-director
             (замысел + динамика) — на каждую главу
/edit        editor: сквозная редактура ВСЕЙ рукописи (межглавные повторы,
             согласованность фактов/эпитетов/заголовков) → EDIT-PASS.md
/maps        cartographer: chapters/<slug>/maps.json → build/build_maps.py
/illustrate  illustrator: PROPOSE (контактный лист) → гейт автора → APPLY
/pdf         build/build_book.py → Vivliostyle → dist/book.pdf (вёрстка)
/proof       вычитка: комментарии человека в PDF (dist/book.annotated.pdf)
             → extract_annots.py → PROOF-<дата>.md → правки revise-loop
/wiki        wiki-curator: непрерывная синхронизация знания + финальный lint
/status      состояние конвейера (читается с диска), что на какой стадии
```

`/proof` — человеко-ведомый цикл (можно повторять): даём PDF → родственник
комментирует в читалке → разбираем и правим точечно. Это «живой документ» из
замысла: уточнения потомков делают книгу полнее. PROOF-файлы — история правок
рода, прежние не стирать (как `wiki/log.md`).

`run.mode: auto` — `/start` гонит всё сам, останавливаясь на гейтах
(иллюстрации, финальный просмотр). `run.mode: staged` — стадии вручную.

**Wiki наполняется С САМОГО НАЧАЛА и НЕПРЕРЫВНО, не в конце.** Это суть
llm-wiki: знание накапливается по мере находок, не теряется. `/ingest` →
стабы людей; `/genealogy` → `wiki/families` + связи; `/research` → по мере
находок в `wiki/units|events|places|people` (через `wiki-curator` после
каждого батча); далее каждая стабилизированная глава → синтез в `wiki`.
`/wiki` — это и **continuous-синхронизация после ingest/genealogy/каждого
research-батча**, и финальный reconcile+lint, а не «один проход в конце».
Если что-то донайдено на любой стадии — сразу в wiki + запись в `wiki/log.md`
(Ingest-workflow §). Перед ответом всегда смотри `wiki/`, не переисследуй.

**Холодный старт: родословной может НЕ быть.** Частый случай — пользователь
почти ничего не знает о предках, документа нет, дерево неизвестно. Норма:
старт от фамилии+села по архивам. **Дерево/дворы — бонус, не предусловие.**
`genealogist` восстанавливает родство из архивов, насколько возможно
(отчество, годы, поле «родственники»/адресат извещения в донесениях, ОБД);
что не восстанавливается — так и фиксирует, **не выдумывает**. Тогда книга
организуется **по людям** (хронология войны / судьба), без дворов; мини-
деревья (`<!-- tree -->`) опускаются без ошибки; эпитеты остаются (по роли/
судьбе). Нам повезло с глубокой родословной — у форкера так не будет; конвейер
обязан работать и «вхолодную».

**Контракт агентов:** каждый перечитывает входы с диска (никакого state в
памяти оркестратора — только файлы), пишет полный артефакт в файл, в reply —
резюме 1–2 экрана. «По периоду X источников не найдено» — нормальная запись,
не дефект. Батчи субагентов — `run.parallel_batch` (3–4, лимиты), по файлу на
агента (без write-конфликтов).

**Проверка делегированной правки (реальные грабли).** Делегированная правка
главы/карты регулярно вносила фактические регрессы и теряла детали. После
любой делегированной правки оркестратор сверяет результат с прошлой версией
(`git diff`): не пропала ли фактура, не появились ли новые утверждения без
источника, правились ли только указанные места. Не уверен — правит точечно
сам, а не принимает молча.

## 4. Конвенции

### Frontmatter (для всех страниц `wiki/`)

```yaml
---
type: person | place | unit | event | family
name: <человекочитаемое имя>
born: <YYYY или YYYY-MM-DD; пусто если неизвестно>
died: <YYYY или YYYY-MM-DD; пусто если жив/неизвестно>
family: <slug ветки>
status: stub | draft | complete | pointer
---
```

Артефакты `book/chapters/<slug>/`: `facts.md`/`context.md` — без frontmatter;
`draft.md` — `revision`, `word_count`, `sources_used`, `status`;
`review.md` — `verdict: ok|revise`, `blockers_count`.

### Имена файлов (slug)

| Тип | Шаблон | Пример |
|---|---|---|
| Люди | `имя-отчество-фамилия` транслит kebab | `ivan-dmitrievich-petrov.md` |
| Места | `название` | `smirnovo.md` |
| Части | `<число>-<тип>[-<число>-<тип>]` | `282-sd.md`, `935-sp-306-sd.md` |
| События | краткий slug | `bagration-1944.md` |
| Семьи | `<фамилия>-<ветка>` | `petrovy-prohor.md` |

Коллизия имён → добавить год рождения: `ivan-petrov-1925.md`.

### Стиль

- Терминология: «семья», «ветка», «фамилия». Не «клан».
- Имена: «<Фамилия>» / «<Фамилия>а» (муж./жен. род).
- Даты: **DD.MM.YYYY** в тексте, ISO в frontmatter. Годы — цифрами, не словами.
- Цитата устной истории — только при наличии конкретной строки-источника,
  с атрибуцией: «Уточнение от <кто>, <дата>:».

### Двусторонний линкинг

Человек упомянут на странице части — на странице части его тоже указать.
Глава ссылается на врезку (не пересказывает её).

## 5. Варианты написания фамилии (методология)

OCR + рукописная дореформенная кириллица искажает фамилии. Перед поиском на
pamyat-naroda **всегда** прогонять все варианты `subject.surname_variants` +
канон. Если список пуст — вывести варианты по правилам и предложить автору:

- **Г ↔ Ч, Г ↔ Б, Г ↔ Л** (петли букв): Каганов→Качанов/Кабанов/Колганов.
- **К ↔ Х** в начале: Каганов→Хаганов.
- **О ↔ А** в безударном слоге: Логунов→Лагунов.
- **У ↔ О** в средней позиции: Логунов→Логонов.
- Близкая, но **другая** фамилия как опечатка (Логунов↔Логинов) — проверять,
  но помечать «возможно другой человек».

pamyat-naroda делает нечёткое сопоставление, но карточки, заведённые **под**
ошибочным написанием, всплывают только при явном поиске по этому варианту —
**запускать оба (все) поиска**. То же для OCR названия села.

## 6. Архивная методология сбора (самая важная стадия)

Сбор данных — фундамент книги. Поверхностный проход = пустая глава или, хуже,
выдумка. Всё ниже — выстрадано реальными промахами; **researcher обязан это
соблюдать целиком**, не «по верхам».

### 6.1. Инструмент и доступ

- **chrome-devtools MCP** — грузит ВАШ локальный Chrome → идёт через вашу
  сеть/VPN (критично для архивов РФ). pamyat-naroda на WebFetch — 403/CSRF.
  **Stagehand/Browserbase не использовать.**
- **Конфликт профиля:** MCP не порулит Chrome, если открыт ваш Chrome на том
  же профиле. Симптом — не цепляется/пусто. → Попросить пользователя закрыть
  Chrome (вернуть просьбу оркестратору) либо запускать отдельным профилем.
- **Не закрывать вкладку/браузер посреди задачи** — пользователь может читать
  страницу. Перед закрытием отдать ссылку оркестратору.
- **Капча / 503 / гео-фенс** (goskatalog, rgakfd, архивы РФ серверно
  недоступны) → **остановиться и вернуть оркестратору просьбу к
  пользователю** (решить капчу / открыть у себя и вставить метаданные). Не
  «повторим позже» молча, не выдумывать.

### 6.2. Протокол карточки человека на pamyat-naroda — ПРОКЛИКАТЬ ВСЁ

Это ядро. Не доверять полям-сводкам — открывать сами документы.

1. **Поиск.** Каждый вариант фамилии (§5) × (село | район | год рождения).
   Фаззи-матч мягкий: карточки, заведённые под опечаткой, всплывают **только**
   при явном запросе по этой опечатке. На выдаче жать «ПОКАЗАТЬ ЕЩЁ» —
   пагинация багует, прокрутить до конца.
2. **Карточка** (`person-hero{ID}`): обойти ВСЕ вкладки/блоки — «Боевой путь»,
   «Документы», «Награды», «Захоронение»; открыть страницу связанной воинской
   части (warunit) для контекста.
3. **Открыть КАЖДЫЙ документ**: наградной лист, приказ о награждении,
   донесение о потерях, картотека ранений, ВПП, УПК, юбилейная карточка,
   запись Книги Памяти. Есть «показать ещё документы» — раскрыть.
4. **Листать страницы документа взад-вперёд** в просмотрщике (стрелки/
   переключатель). Документ почти всегда многостраничный: наградной =
   представление + оборот + приказ; донесение = несколько листов. Собрать ВСЕ
   страницы **этого** документа. ⚠ Если счётчик показывает > ~2000 страниц —
   вы соскользнули в общую подшивку с **чужими людьми**: на каждой странице
   проверять, что ФИО/часть — про ЭТОГО человека. *«Будь аккуратен, чтоб не
   набрать лишнего.»*
5. **Скачать сканы** нужных документов в `assets/documents/<slug>/` и дописать
   `assets/documents/<slug>/MANIFEST.md` (тип документа · архив/фонд/опись/
   дело/лист · дата · URL · чей · к какому факту). CDN
   `cdnc.pamyat-naroda.ru/{podvig|memorial}/…/00000NNN.jpg` отдаёт картинку
   **только** с `Referer` страницы героя + `User-Agent`; без `Referer` —
   заглушка ~190 байт; браузерный `fetch` CORS-блокирован → качать `curl` с
   этими заголовками (или через сам браузер). Это «отдельно собранные
   документы»: в книгу-иллюстрации они НЕ идут (их подбирает `illustrator`),
   но это первоисточник фактов и часто портретов.
6. **DOM-подсказки** (для `evaluate_script`): карточки документов —
   `.js-hero-card-doc` (атрибуты `data-doc-type`, `data-doc-id`,
   `data-doc-pages` — JSON со списком страниц); просмотрщик —
   `.js-imageviewer-controls-center-item`. URL: `/heroes/person-hero{ID}/`,
   `…-podvig-chelovek_nagrazhdenie{ID}`, `…-memorial-chelovek_donesenie{ID}`,
   `…-memorial-chelovek_pechatnoi_knigi_pamyati{ID}`, `…-gospital{ID}`,
   `kld-card_ran{ID}`. Расширенный поиск (итерировать варианты программно):
   `…/?last_name=<вариант>&place_birth=<село/район>&group=all&types=<все
   типы документов>&grouppersons=1` — затем «ПОКАЗАТЬ ЕЩЁ» до конца.

### 6.4. Рассуждение — не потерять (это были реальные ошибки)

- **Сначала зафиксировать находки**, потом следующий запрос/перепроверка.
- **Чей это лист.** Справка может физически лежать рядом с ЧУЖИМ приказом в
  одной подшивке — не приписывать часть/награду по соседнему скану
  (реальный промах: «6 гв ВДСП» из соседней справки).
- **Награды:** «За отвагу (2)» = упомянут в 2 ДОКУМЕНТАХ, не 2 медали.
  Считать по приказам о награждении, дубли-сканы дедуплицировать. Не путать
  «фигурирует в N документах» с «N наград».
- **Номера** части/армии, № партбилета — строго из документа, перепроверять
  (реальные промахи: 19 А vs 39 А, неверный № партбилета).
- **Родство** однофамильцев из других сёл — подтверждать совпадением отчества
  + года рождения + привязки к селу; иначе «возможно не наш».
- **Тонкие линии не выдумывать:** мало данных → один стаб, не сочинять людей.
- **Конфликт дат/фактов** — записать ОБЕ версии (приоритет: уточнение родни >
  архив > OCR), не выбирать молча и без драмы (§3).
- В `facts.md` сверки дописывать append-only блоком
  `## АРХИВНАЯ СВЕРКА <дата>` — прежнее не стирать.
- Оркестратор **проверяет вывод `researcher`** прежде чем доверять (классы
  ошибок выше). Это явная стадия `/research`, не формальность.

## 7. Дисциплина сборки и память

- **Пересборка PDF — только по явной команде** пользователя («пересобери»).
  Не пересобирать самовольно после каждой правки.
- `build/clip_geo.py` скачивает Natural Earth и режет по `geo:` из конфига —
  запускать один раз после заполнения конфига (см. README).
- **Memory** (`~/.claude/projects/<...>/memory/`) — метазаметки об ЭТОМ
  проекте/пользователе (что любит, как отвечать). Не дублирует wiki. Wiki —
  факты о роде. В новой сессии: `book.config.yml` → `wiki/index.md` →
  `MEMORY.md` → этот файл.
